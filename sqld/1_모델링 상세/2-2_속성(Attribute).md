# 🌟 1단계 특별 Q&A: '취미'가 3개인 사람은 어떻게 저장하나요? (원자성과 다중값의 딜레마)

SQLD에서 속성(Attribute) 파트를 공부하는 입문자들이 가장 많이 던지는, 그리고 실무에서도 아주 빈번하게 마주하는 근본적인 질문이 있습니다.

*"속성은 반드시 '하나의 값(단일 값)'만 가져야 한다고 배웠습니다. 그런데 어떤 회원이 '독서, 등산, 영화감상' 이렇게 3개의 취미를 가지고 있으면 어떻게 하죠? 하나의 칸에 콤마(,)로 적으면 안 된다고 하니, 아예 테이블에 `[취미1]`, `[취미2]`, `[취미3]` 처럼 칸을 3개로 미리 만들어두면 안 되나요?"*

정말 훌륭한 접근입니다! 이 질문에 대한 명확한 해답을 통해 데이터 모델링의 원리를 완벽하게 이해해 봅시다.

---

## 1. ❌ 첫 번째 오답: 콤마(,)로 한 칸에 몰아넣기
가장 초보적인 실수이며, 제1정규화(1NF)를 정면으로 위반하는 행위입니다.

* **상황:** `[취미]` 칸에 "독서, 등산, 영화감상" 이라고 입력합니다.
* **치명적 문제:** 나중에 "취미가 '등산'인 회원만 검색해 줘!" 라고 데이터베이스에 명령(`SELECT`)할 때 엄청난 문제가 생깁니다. 컴퓨터는 칸 전체의 글자를 하나하나 분리해서 검사해야 하므로 검색 속도가 끔찍하게 느려지며(인덱스 사용 불가), 통계를 내기도 불가능에 가깝습니다.

## 2. ❌ 두 번째 오답: 칸(속성)을 여러 개로 쪼개기
질문자님이 떠올리신 방법입니다. `[취미1]`, `[취미2]`, `[취미3]` 으로 속성을 늘려버리는 방식입니다. 언뜻 보면 콤마(,)의 문제를 해결한 것 같지만, 데이터베이스 관점에서는 여전히 '나쁜 설계'입니다.

* **치명적 문제 1 (공간 낭비):** 취미가 아예 없는 회원은 저 3개의 칸이 전부 빈칸(`NULL`)으로 버려지게 됩니다. 데이터베이스에서 빈칸이 많다는 것은 공간 낭비이자 성능 저하의 원인입니다.
* **치명적 문제 2 (확장성 부족):** 만약 취미가 4개인 열정적인 회원이 가입하려고 하면 어떡할까요? 테이블의 구조(설계도) 자체를 뜯어고쳐서 `[취미4]` 칸을 새로 만들어야 하는 대참사가 일어납니다.
* **치명적 문제 3 (검색의 어려움):** '등산'을 좋아하는 사람을 찾으려면 `WHERE 취미1 = '등산' OR 취미2 = '등산' OR 취미3 = '등산'` 처럼 조건을 무식하게 길게 적어야 합니다.

## 3. 💡 정답: 새로운 '엔터티(테이블)'로 독립시키기!
데이터 모델링에서 **"하나의 속성이 여러 개의 값(다중값)을 가질 수밖에 없는 운명"**이라면, 그것은 더 이상 단순한 속성이 아닙니다. **"나도 어엿한 엔터티(테이블)로 독립시켜 줘!"**라는 강력한 신호입니다.

* **올바른 설계:** '회원' 테이블에서 취미 속성을 아예 지워버립니다. 대신 **'회원_취미'라는 새로운 테이블**을 하나 만듭니다.
* **데이터 저장 방식:**
  * 1번 회원 - 독서
  * 1번 회원 - 등산
  * 1번 회원 - 영화감상
* **효과:** 이렇게 데이터를 세로(행)로 늘어나게 만들면, 취미가 100개인 사람이 와도 칸을 새로 만들 필요 없이 데이터만 한 줄씩 추가하면 끝입니다. 취미가 없는 사람은 아예 데이터를 안 넣으면 되니 빈칸(`NULL`)이 생길 일도 없습니다. 검색 속도 역시 월등히 빨라집니다.

---

## 🎯 최종 요약

* **원자성(하나의 값만 가질 것)**은 절대 타협할 수 없는 데이터베이스의 제1원칙입니다.
* 가로(열/칸)로 늘리려는 유혹을 참으세요. 관계형 데이터베이스는 가로(칸)가 늘어나는 것보다, **세로(행/줄)가 늘어나는 것을 훨씬 좋아하고 잘 처리하도록 설계**되어 있습니다.
* 속성에 값이 여러 개 들어갈 것 같다면? 주저하지 말고 **테이블을 쪼개서(정규화) 새로운 엔터티로 분리**하세요!
