# 🔍 1단계 심화: 데이터 모델링의 '용어 족보' (논리 vs 물리)

데이터 모델링을 공부하다 보면 "엔터티라고 했다가 테이블이라고 했다가, 대체 뭐가 다른 거죠?"라는 의문이 듭니다. 이 둘은 사실 같은 대상을 의미하지만, **'어느 단계에서 부르느냐'**에 따라 이름이 달라집니다. 이를 확실히 구분하는 것은 SQLD 1과목의 기본기입니다.

---

## 1. 📂 용어 매칭 테이블

| 구분 | 논리적 모델링 (설계 단계) | 물리적 모델링 (구축 단계) | 엑셀(Excel) 비유 |
| :--- | :--- | :--- | :--- |
| **개체 전체** | **엔터티 (Entity)** | **테이블 (Table)** | 엑셀 시트 (Sheet) |
| **항목 정의** | **속성 (Attribute)** | **컬럼 (Column)** | 열 제목 (A, B, C...) |
| **개별 데이터** | **인스턴스 (Instance)** | **레코드/로우 (Record/Row)** | 행 (데이터 1줄) |
| **고유 식별값** | **주식별자 (Identifier)** | **기본키 (Primary Key, PK)** | 고유 일련번호 |
| **참조 관계값** | **외부식별자 (Foreign ID)** | **외래키 (Foreign Key, FK)** | VLOOKUP 참조값 |

---

## 2. 💡 단계별 상세 설명

### 🏗️ 논리적 모델링 (Logical Modeling)
* **언제:** 사용자와 시스템 분석가가 모여 "어떤 데이터를 어떻게 관리할까?"를 고민하며 **화이트보드에 그림(ERD)을 그릴 때** 사용합니다.
* **성격:** 컴퓨터 환경(오라클, MySQL 등)에 종속되지 않는, 순수한 비즈니스 로직 중심의 개념입니다.
* **사용 언어:** "이 **엔터티**에 어떤 **속성**들이 필요할까요?"

### ⚙️ 물리적 모델링 (Physical Modeling)
* **언제:** 설계도를 바탕으로 실제 컴퓨터에 **데이터베이스 소프트웨어를 설치하고 `CREATE TABLE` 명령어를 치며 공간을 만들 때** 사용합니다.
* **성격:** 특정 데이터베이스 제품의 특성을 고려하여, 저장 공간과 속도를 최적화하는 구현 단계입니다.
* **사용 언어:** "이 **테이블**의 이 **컬럼**은 `VARCHAR(20)` 타입으로 설정합시다."

---

## 3. 🚨 SQLD 시험 대비: 주의해야 할 함정!

시험 문제에서는 이 둘을 교묘하게 섞어서 오답을 유도합니다. 다음 문장을 보며 올바른지 판단해 보세요.

> **"논리적 데이터 모델링 단계에서는 테이블명과 컬럼의 데이터 타입(INT, VARCHAR 등)을 상세히 정의해야 한다."**

* **정답:** **X (틀림!)**
* **해설:** 데이터 타입(INT, VARCHAR 등)이나 테이블명과 같은 **물리적인 기술 정보**는 논리적 모델링 단계가 아니라, **물리적 데이터 모델링 단계**에서 정의하는 것입니다. 
* 논리적 모델링 단계에서는 오직 **'업무적으로 의미 있는 엔터티와 속성의 명칭'**에만 집중해야 합니다.

---

## 🎯 요약: 용어의 짝꿍을 기억하세요!
데이터를 설계할 때는 **[엔터티 - 속성 - 인스턴스]**라는 이름을 쓰고, 실제로 구현해서 데이터를 담을 때는 **[테이블 - 컬럼 - 로우]**라는 이름을 쓴다고 생각하면 절대 헷갈릴 일이 없습니다!
