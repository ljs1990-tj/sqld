# 📖 1과목 핵심 요약: 데이터 독립성과 3단계 스키마 구조

## 1. 🛡️ 데이터 독립성: "충격 흡수 장치"
건물의 1층 인테리어를 바꾼다고 해서 2층 바닥이 무너지면 안 되듯, **하위 단계를 수정하더라도 상위 단계 프로그램이 고장 나지 않도록 방어하는 성질**을 '데이터 독립성'이라고 합니다.

* **① 논리적 독립성 (개념 스키마 ↔ 외부 스키마 방어)**
  * **상황:** DBA가 학교 시스템에 '기숙사'라는 새로운 테이블(구조)을 추가하거나 기존 구조를 변경했습니다.
  * **효과:** 구조가 바뀌어도 학생들이 로그인해서 보는 '웹 수강신청 화면(응용프로그램)'은 전혀 에러 없이 정상 작동합니다.
* **② 물리적 독립성 (내부 스키마 ↔ 개념 스키마 방어)**
  * **상황:** 학교 서버실의 구형 하드디스크를 최신형 SSD로 교체하거나 데이터 저장 위치를 바꿨습니다.
  * **효과:** 물리적인 저장 장치가 바뀌어도, DBA가 짜놓은 테이블 설계도(개념 스키마)나 학생들의 수강신청 화면에는 아무런 영향이 없습니다.

## 2. 🏢 데이터베이스 3단계 구조 (3-Schema)
데이터 독립성을 지키기 위해 ANSI(미국국립표준협회)에서 데이터베이스를 바라보는 관점을 3개의 층으로 쪼개 놓은 것입니다. (스키마 = 설계도)

### ① 외부 스키마 (External Schema) - "사용자 관점" (여러 개 존재)
* **비유:** 학생들이 스마트폰으로 보는 **[내 수강 시간표]** 화면, 또는 교수님이 보는 **[내 수업 출석부]** 화면입니다.
* **특징:** 전체 DB 중 사용자에게 필요한 아주 일부분만 보여줍니다. 보는 사람에 따라 뷰(View)가 다르기 때문에 **여러 개**가 존재합니다.

### ② 개념 스키마 (Conceptual Schema) - "DBA/통합 관점" (단 1개 존재)
* **비유:** 총장님과 DBA가 가지고 있는 **'학교 전체 데이터의 마스터 설계도'**입니다.
* **특징:** 학생, 과목, 수강내역 등 모든 테이블과 규칙이 다 모여있는 거대한 전체 밑그림입니다. DB 전체에 **단 1개**만 존재하며, 보통 "스키마 그렸어?"라고 할 때 이 개념 스키마(ERD)를 말합니다.

### ③ 내부 스키마 (Internal Schema) - "시스템/물리적 관점"
* **비유:** 서버실 엔지니어가 쳐다보는 **'0과 1, 그리고 하드디스크'**입니다.
* **특징:** 데이터가 실제로 컴퓨터 하드웨어의 어느 구역에, 어떤 크기로, 어떻게 저장될지를 다루는 가장 밑바닥 단계입니다.

## 3. 🎨 데이터 모델의 표기법 (ERD)
"학생이 과목을 수강한다"는 업무 규칙을 전 세계 공통의 그림(ERD)으로 그리는 방법입니다.

### ① Chen (첸) 표기법
* **특징:** 관계를 마름모(◇)로 표현합니다. 최초의 표기법이지만 실무에서는 거의 쓰지 않고 이론에서만 다룹니다.
* **수강신청 비유:** `[학생] ── ◇수강한다◇ ── [과목]`

### ② 🌟 IE / Crow's Foot (까마귀 발) Barker / UML 표기법
* **특징:** 현재 실무와 SQLD 시험에서 **가장 압도적으로 많이 쓰이는 표준 표기법**입니다. 데이터가 여러 개(Many) 연결될 수 있는 쪽에 **'까마귀 발(삼지창)'** 모양을 그립니다.
* **수강신청 비유:**
  * 1명의 학생은 '여러 개'의 수강내역을 가질 수 있습니다.
  * 따라서 `학생` 쪽은 작대기 하나(`|`)를 긋고, `수강내역` 쪽에는 까마귀 발(`∈`)을 그립니다.
  * **표현:** `[학생] ───+-----------< [수강내역]` (학생 1명이 여러 수강내역을 가짐)

* **③ Barker (바커) 표기법:**
  * **특징:** 오라클(Oracle)에서 표준으로 채택하여 사용하는 표기법입니다. 선에 동그라미(O)나 까마귀 발 모양을 결합해서 그립니다.
* **④ UML 표기법:**
  * **특징:** 데이터베이스보다는 소프트웨어(객체지향) 설계에서 주로 쓰는 방식입니다. 표에 `<<Entity>>` 같은 태그를 달아 표현합니다.

### 💡 실전 대비 기출 포인트
* "사용자 관점, 여러 개 존재한다" ➔ **외부 스키마**
* "모든 데이터 구조를 묶은 전체 데이터베이스, DBA 관점" ➔ **개념 스키마**
* "저장 장치 구조 변경이 응용프로그램에 영향을 주지 않는다" ➔ **물리적 독립성**
* "관계의 다(Many)를 나타낼 때 까마귀 발을 쓴다" ➔ **IE 표기법**
